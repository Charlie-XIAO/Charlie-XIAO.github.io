ENH: When merging with incompatible keys, include the first incompatible key in the error message

Table of Contents

  • The Issue
  • The Approach

Hi, welcome to my first post on open source contributions. This post is about my contribution to the powerful Python data analysis toolkit, pandas. This is a only small code contribution that improves some error message, but it is definitely an interesting and encouraging experience.

  • Project: pandas.
  • Relevant issues: #51861.
  • Relevant pull requests: #51947.



The Issue

It all started from the issue ENH: More helpful error messages for merges with incompatible keys. The author of the issue asked for more helpful messages to indicate the incompatible keys when pd.merge raises an exception. He provided a simple examples as follows: 

df_left = pd.DataFrame([
    {"a": 1, "b": 1, "c": 1, "d": 1, "z": 1},
])
df_right = pd.DataFrame([
    {"a": 1, "b": 1, "c": 1, "d": None, "z": 1},
])
merge = pd.merge(df_left, df_right, indicator="source")

Clearly, the key d is incompatible, since in df_left it is of numerical type but in df_right it is of object type. However, the initial error message simply reports that one is trying to merge on int64 and object columns, without specifying the incompatible key(s). This can be confusing especially if there are a lot of columns, since a user would need to look through every single column to find out the incompatible key. Therefore, the maintainers agreed that this is a reasonable feature request.



The Approach

The first important point is to determine how and to what extent I would make the enhancement. I could have traversed all the columns of the dataframes and report all incompatible columns, but that would be inefficient, and could make the error message too long. Alternatively, I chose to only report the first incompatible column. This should be sufficient to solve the issue, since one can always change the first incompatible column and try merging again. Then the next incompatible column, if exists, would be reported.

The next step was to make unit tests, according to the principles of test-driven development. I searched for the error messages in the repository via git grep, and found that pd.join and pd.merge can both raise that error message. The original test framework already included the checks for this error message, but they used the minimal test cases with dataframes consisting of only a single column. In order to make sure that the reported incompatible key is correct, I made additional test cases with dataframes consisting of multiple columns, and merging on different column(s). Note that using pytest, the error message checking can be done such as 

msg = "Your error message"
with pytest.raises(Exception, match=msg):  # Can also be more specific, e.g. ValueError
    merge(df1, df2, on=["A", "B"])

Finally, it's time to struggle to pass the tests that I've made. This is in fact a fairly simple task. Notice that in the function that the error is raised, there is a for loop such that 

for lk, rk, name in zip(
    self.left_join_keys, self.right_join_keys, self.join_names
):
    ...

Clearly lk is the entry from the left dataframe, rk is the entry from the right dataframe, and name would be the name of the column. An error would be raised whenever the types of lk and rk are incompatible. Since we are only reporting the first incompatible key, using name would be sufficient without any additional traversal. Therefore, it would essentially be a one-line fix for each place where this error is raised, such that 

    msg = (
-       f"You are trying to merge on {lk.dtype} and "
-       f"{rk.dtype} columns. If you wish to proceed you should use pd.concat"
+       f"You are trying to merge on {lk.dtype} and {rk.dtype} columns "
+       f"for key '{name}'. If you wish to proceed you should use pd.concat"
    )

After making the pull request, I added the corresponding changelog, and the pull request got merged shortly after that. This contributing experience is pretty smooth without really running into any problems. This is also my first contribution to pandas, and I'm looking forward to contributing more in the future.
