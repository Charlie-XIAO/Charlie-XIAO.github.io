Groupby

BUG: groupby sum turning inf+inf and (-inf)+(-inf) into nan #53623
Previously when calling sum on a GroupBy object, it summed inf+inf and (-inf)+(-inf) to nan, which is undesired. Moreover, this behavior was inconsistent with calling apply on a standard summation function. The problem was caused by the Cython function group_sum, which implements Kahan's summation to reduce numerical error caused by finite-precision floating-point operations. It maintains a compensation for low-order bits, and fix the excess in further rounds, which can be interpreted as follows. 
var y = input[i] - c
var t = sum + y
c = (t - sum) - y    // c is initialized to zero before the loop
sum = t              // sum is initialized to zero before the loop
next i
In the case where input is inf, then y and t would become inf, and thus when computing c, i.e., the compentation, we would be performing inf-inf which gives nan. To fix this, we can set the compensation back to zero whenever it becomes nan. Also note that for efficiency, this Cython function is written with nogil so that we cannot use util.is_nan. Instead, we can use c != c to represent util.is_nan(c).
FIX groupby with column selection not returning tuple when grouping by list of a single element #53517
When we traverse a GroupBy object like for k, v in df.groupby(["a"]), even though ["a"] is a list of a single element, the returned keys are tuples. However, after column selection such as for k, v in df.groupby(["a"])["a"] or for k, v in df.groupby(["a"])[["a"]], the returned keys k were no longer tuples. This is undesired, and caused by keys getting lost from the very beginning when creating the GroupBy objects. With @rhshadrach's help, I fixed this by passing keys explicitly instead of using groupers to imply keys. Now the behaviors with and without column selection are consistent.
BUG DataFrameGroupBy.agg with list not respecting as_index=False #53237
When groupby is called with as_index=False, the group labels of the aggregated output should be columns rather than index, which is effectively SQL-style. However, if the aggregation functions are passed in as a list, as_index is not respected. I modified the logic of result processing so that this bug is fixed.
BUG: GroupBy.quantile implicitly sorts index.levels #53049
Previously, even if groupby is called with sort=False, the indices may get sorted. For instance, 
>>> ind = pd.MultiIndex.from_tuples(
...     [(0, "B"), (0, "A"), (1, "B"), (1, "A")],
...     names=["sample", "category"],
... )
>>> ser = pd.Series(range(4), index=ind)
>>> ser  # Here, we shall see that the B's are all before the A's
sample  category
0       B           0
        A           1
1       B           2
        A           3
dtype: int64
>>> grp = ser.groupby(level="category", sort=False).quantile([0.2, 0.8])
>>> grp.unstack()  # Here, we can see that A and B are sorted though sort=False
          0.2  0.8
category          
A         1.4  2.6
B         0.4  1.6
After investigating, this is caused by index.levels being sorted, and some methods are using those levels. This does not happen for MultiIndex since the index levels are manually created, so we manually create for non-multi Index as well. Then we found an approximately 50% performance regression due to always using 64-bit indices. We then called coerce_indexer_dtype to avoid this regression.

I/O

BUG: complex Series/DataFrame display all complex nans as nan+0j #53844
This is related to pandas-dev/pandas#53764. Previously in that PR (also made by me), I considered nan of complex dtype simply as nan+0.0j, but this is incorrect. Both the real and imaginary part can be nan, and either of these parts being nan will lead to util.is_nan returning True. Therefore, the previous PR led to all types of complex nans being displayed as NaN+0.0j, though the underlying data is stored correctly. In this PR, I split real and imaginary parts, format them separately, and concatenate them (hopefully) properly. Since the imaginary part can also be NaN, it also has to be padded manually to the maximum length. For instance, we may have 
0      NaN-1.2345j
1      NaN+   NaNj
2  -1.2345+   NaNj
BUG: bad display for complex series with nan #53764
This is related to pandas-dev/pandas#53862. Though complex("nan") no longer raises in series and dataframe, it is not displayed correctly, for instance N000a000N. This is also because pandas not considering that complex numbers can have nans, thus splitting by +, -, and j and maunally adding zeros for padding, causing this issue to happen. I applied some tricks here: if dtype is complex then I do not display nans as NaN but as NaN+0.0j. The I split the real and imaginary parts with some regex and pad both the real and imaginary parts of a single number but also all real (resp. imaginary) parts of all the numbers using some existing helper function. Now complex("nan") can be displayed properly, for instance, pd.Series([1.23, complex("nan"), -1.2j]) will print 
0   1.23+0.00j
1    NaN+0.00j
2   0.00-1.20j
dtype: complex128
BUG: MultiIndex displays incorrectly with a long element #53044
I/O formatting was failing to display MultiIndex objects with a long elements, such as 
print(pd.MultiIndex.from_tuples([("c" * 62,)]))  # Raises an error
This is because an important variable during I/O formatting was set only when breaking out of a for loop. It did not consider cases where the for loop terminates naturally, causing further errors. I fixed this bug so that such objects can be displayed correctly.

Numeric

BUG: series with complex nan #53682
Previously, series and dataframe with complex("nan") will raise an error. This is caused by a hard conversion from complex nan into float dtype in the Cython code. When designing the constructor, pandas considered all objects that util.is_nan evaluate to True as float nans. I added a check for complex nans to avoid this invalid conversion. complex("nan") is now allowed in series and dataframe, displayed as NaN+0.0j.
FIX RangeIndex rsub by constant #53288
Previously, negative RangeIndex correctly set the step size to the opposite of the original, but constant minus RangeIndex did not. This was because the former would call __neg__ while the latter would undergo some more complex numeric operations. I made a small modification so that now the latter case also reverts the step size correctly.

Resample

BUG: resampling empty series loses time zone from dtype #53736
For instance, look at the following code snippet. 
df = pd.DataFrame({"ts": [], "values": []}).astype({"ts": "datetime64[ns, Atlantic/Faroe]"})
result = df.resample("2MS", on="ts", closed="left", label="left", origin="start")["values"].sum()
print(result.index)  # This prints DatetimeIndex([], dtype='datetime64[ns]', name='ts', freq='2MS')
Clearly the timezone information is lost, but this does not happen if the constructing series is not empty. The reason for this bug is that, the case of empty series is dealt with separately and dtype is not explicitly provided. Therefore, pandas tried to provide a minimum compatible time for the empty index, causing the loss of timezone. I made an easy fix that takes dtype into consideration.

Reshaping

BUG Merge not behaving correctly when having MultiIndex with a single level #53215
Suppose that we have two dataframes as follows. 
>>> df = pd.DataFrame({"col1": ["A"]})
>>> df
col1
0    A
>>> df2 = pd.DataFrame(
...     data={"col2": [100]},
...     index=pd.MultiIndex.from_tuples([("A",)], names=["col1"]),
... )
      col2
col1      
A      100
>>> df2
Then when we write 
df.merge(df2, left_on=["col1"], right_index=True, how="left")
we would expect a single-row dataframe with column col1 as A and column col2 as 100. But previously, the column col2 would be the missing value NaN, which is undesired. This is because when there is a single join key, pandas would treat as indices as single index, thus "A" and ("A",) were treated as different keys. I modified this part of logic to treat MultiIndex correctly and simplified the code.
ENH include the first incompatible key in error message when merging #51947
DataFrame.merge raises an error when there are imcompatible keys, but there was no information provided in the error message that indicate which keys are incompatible. I improved the error message by displaying the first imcompatible key. For instance, 
ValueError: You are trying to merge on int64 and object columns for key 'd'. If you wish to proceed you should use pd.concat
