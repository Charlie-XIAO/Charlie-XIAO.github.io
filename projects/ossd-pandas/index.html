Groupby

BUG: groupby sum turning inf+inf and (-inf)+(-inf) into nan #53623
Previously when calling sum on a GroupBy object, it summed inf+inf and (-inf)+(-inf) to nan, which is undesired. Moreover, this behavior was inconsistent with calling apply on a standard summation function. The problem was caused by the Cython function group_sum, which implements Kahan's summation to reduce numerical error caused by finite-precision floating-point operations. It maintains a compensation for low-order bits, and fix the excess in further rounds, which can be interpreted as follows. In the case where input is inf, then y and t would become inf, and thus when computing c, i.e., the compentation, we would be performing inf-inf which gives nan. To fix this, we can set the compensation back to zero whenever it becomes nan. Also note that for efficiency, this Cython function is written with nogil so that we cannot use util.is_nan. Instead, we can use c != c to represent util.is_nan(c).
FIX groupby with column selection not returning tuple when grouping by list of a single element #53517
When we traverse a GroupBy object like for k, v in df.groupby(["a"]), even though ["a"] is a list of a single element, the returned keys are tuples. However, after column selection such as for k, v in df.groupby(["a"])["a"] or for k, v in df.groupby(["a"])[["a"]], the returned keys k were no longer tuples. This is undesired, and caused by keys getting lost from the very beginning when creating the GroupBy objects. With @rhshadrach's help, I fixed this by passing keys explicitly instead of using groupers to imply keys. Now the behaviors with and without column selection are consistent.
BUG DataFrameGroupBy.agg with list not respecting as_index=False #53237
When groupby is called with as_index=False, the group labels of the aggregated output should be columns rather than index, which is effectively SQL-style. However, if the aggregation functions are passed in as a list, as_index is not respected. I modified the logic of result processing so that this bug is fixed.
BUG: GroupBy.quantile implicitly sorts index.levels #53049
Previously, even if groupby is called with sort=False, the indices may get sorted. For instance, After investigating, this is caused by index.levels being sorted, and some methods are using those levels. This does not happen for MultiIndex since the index levels are manually created, so we manually create for non-multi Index as well. Then we found an approximately 50% performance regression due to always using 64-bit indices. We then called coerce_indexer_dtype to avoid this regression.

Numeric

BUG: series with complex nan #53682
Previously, series and dataframe with complex("nan") will raise an error. This is caused by a hard conversion from complex nan into float dtype in the Cython code. When designing the constructor, pandas considered all objects that util.is_nan evaluate to True as float nans. I added a check for complex nans to avoid this invalid conversion. complex("nan") is now allowed in series and dataframe, displayed as NaN+0.0j.
FIX RangeIndex rsub by constant #53288
Previously, negative RangeIndex correctly set the step size to the opposite of the original, but constant minus RangeIndex did not. This was because the former would call __neg__ while the latter would undergo some more complex numeric operations. I made a small modification so that now the latter case also reverts the step size correctly.

Reshaping

BUG: pd.concat dataframes with different datetime64 resolutions #53641
Previously, pandas raises a ValueError complaining about shape mismatch if we concatenate two datetime dataframes with different time resolutions. Looking into the code base, datetime dataframes with different time resolutions fall in the same branch as extension array types. However, extension array types have by default axis=1 upon concatenation, while datetime dataframes have axis=0. Therefore, the axis keyword must be explicitly passed to the _concat_same_type method of the corresponding class. Yet some classes are not supporting the axis keyword in that method, so I had to implement a dirty workaround distinguishing the cases. The outcome is nice: it solved its original issue and several other issues that arose during its review process, including implicit concatenations such as df.loc[1] = ... to a single-row dataframe. Now concatenations work smoothly for datetime dataframes with different time resolutions.

Others

BUG: avoid DeprecationWarning when the Series has index as list of Series #55395
Series were unexpectedly raising a deprecation warning when index is a list of Series. This is because when validating index type, pandas is implicitly checking for the _data attribute, which has been deprecated for some classes. Instead, I made it to check for only certain classes, but since the check happens in Cython code, it suffered from circular import. I used runtime importing to resolve this issue and checked that there is no significant performance degradation. The Series constructor should now work without deprecation warnings, and accept only valid types of index.
